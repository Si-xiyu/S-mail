# SmartMail 项目架构与技术选型

本文是 SmartMail 项目的主入口文档。新成员或 Agent 进入项目时，应先阅读本文，再按需要跳转到 `docs/` 下的展开文档。

## 1. 产品定位

SmartMail 的目标不是传统邮箱加几个 AI 按钮，而是一个类 Notion Mail 的 AI 原生邮件工作台。

核心体验是：

- 邮件进入系统后自动完成摘要、分类和 Junk 判断。
- 用户打开邮件时，AI 摘要默认显示在正文前。
- 左侧工作区提供 View、Mail、用户分类、系统功能、头像和 Settings。
- 中栏展示邮件列表，强调可扫描性：摘要首句、分类、状态、附件标记、优先级。
- 点击邮件后右侧抽屉展开详情，展示摘要卡片、正文、附件和当前邮件 Agent 入口。
- 左侧系统功能提供全局 Agent 入口，MVP 中可先通过 mock RAG Tool 占位，MVP 后接入真实全量邮箱问答。

课程要求允许不使用真实 SMTP/POP3，且明确“假设客户端在同一个服务器下，不考虑服务器间的传递”。因此第一版不做外部邮箱通信，不做完整邮件协议服务器。

## 2. 范围边界

### 2.1 MVP 必须交付

- SmartMail 系统内邮箱注册、登录和鉴权。
- 同一 SmartMail 服务器内用户之间收发邮件。
- 收件箱、已发送、草稿、垃圾箱、Junk 等基础视图。
- 本地草稿自动保存：优先保证同一浏览器内恢复。
- 附件上传和下载：选择文件即上传为 Pending Attachment，发送时绑定为正式附件。
- 邮件正文与附件分离存储：正文入库，附件文件存磁盘，数据库保存附件元信息。
- 用户视角软删除：删除只影响当前用户的 Mailbox Item，不影响其他用户。
- AI 自动分析管道：投递后异步生成摘要、分类、Junk 判断和分析状态。
- 当前邮件 Agent：从右侧邮件详情抽屉呼出，默认以当前邮件为上下文。
- 工作台式 UI：前端体验是硬目标，不是最后的美化项。

### 2.2 MVP 后必须追加

- 全量邮箱问答能力。
- RAG Tool 真实实现：BM25 + 向量检索 + RRF 融合排序。
- 向量检索异常时自动降级为 BM25。
- 本地 Ollama 小模型用于 embedding，主问答模型使用 DeepSeek API，二者配置分离。
- 草稿本地 + 服务器同步。
- SSE 或 WebSocket 新邮件通知。

### 2.3 明确不进入第一版

- 外部邮箱地址真实通信。
- SMTP/POP3/IMAP 主链路。
- Redis、消息队列、向量库作为必需依赖。
- 附件在线预览、病毒扫描、超大文件断点续传。
- AI 分析阻塞邮件投递。

## 3. 架构总览

```text
Vue 3 AI Mail Workspace
    ↓ HTTP REST / Workspace View API
Spring Boot 3 主业务后端
    ├─ MySQL 或 H2 dev profile
    ├─ 本地附件文件存储
    ├─ AI 分析任务表 / 后台轮询
    ├─ Workspace API
    └─ Internal Tool API
            ↓ HTTP
Python FastAPI Agent 服务
    ├─ Automatic Analysis Pipeline
    ├─ Interactive Tool Router
    ├─ RAG Tool，MVP mock
    ├─ 规则 fallback
    └─ DeepSeek / Ollama，MVP 后逐步接入
```

关键原则：

- 前端只调用 Spring Boot API，不直连 Agent 或数据库。
- Agent 不直连数据库，通过后端 Internal Tool API 访问业务数据。
- 邮件投递同步完成，AI 分析异步补齐。
- 自动分析管道和交互式 Tool Router 分离。
- 核心存储边界是主数据库 + 文件存储；Redis、消息队列、向量库都是增强依赖。
- 前端主体验使用 Workspace View Model，不按数据库表 CRUD 拼页面。

## 4. 核心领域模型

领域术语以 [CONTEXT.md](./CONTEXT.md) 为准。关键概念如下：

- **SmartMail System Mailbox**：同一 SmartMail 服务器内注册的邮箱账号。
- **External Mailbox**：系统外邮箱，是扩展方向，不进入 MVP 主链路。
- **Draft**：用户尚未发送的邮件内容，只属于创建者。
- **Local Autosave**：同一浏览器内自动保存草稿。
- **Pending Attachment**：已上传但尚未绑定到正式邮件的附件。
- **Attachment**：随正式邮件发送的文件，文件内容与邮件正文分离存储。
- **Mailbox Item**：某个用户对一封已投递邮件的私有视图。
- **Mailbox Folder**：Inbox、Sent、Trash、Junk 等系统级位置。
- **Mail Category**：用户私有的语义分类，和文件夹分开建模。
- **Automatic Analysis Pipeline**：后台自动摘要、分类、Junk 判断。
- **Interactive Tool Router**：用户主动调用 Agent 时的工具路由和权限策略。
- **Workspace View Model**：为前端工作台体验设计的 API 数据形状。

## 5. 关键产品决策

### 5.1 文件夹与分类分离

文件夹表示系统位置，例如 Inbox、Sent、Trash、Junk。分类表示语义分组，例如课程、会议、项目、账单、Other。

普通 AI 分类不改变文件夹；Junk 是特例，用户体验上应从普通 Inbox 分流到 Junk 视图。

### 5.2 自动分析不阻塞投递

发送邮件后，收件人应立即能看到邮件。AI 分析状态可以是 pending、succeeded、failed。摘要和分类完成后再补齐到 Workspace View Model。

### 5.3 两条 AI 执行通道

自动分析管道：

- 后台系统任务触发。
- 只允许写摘要、分类、Junk、分析状态等有限结果。
- 不需要用户逐次确认。

交互式 Tool Router：

- 用户主动触发。
- 默认只读。
- 写操作必须用户确认。
- 用户可在 Settings 中开启自动写操作，但这是高级开关。

### 5.4 RAG Tool 先 mock 后替换

MVP 中保留 RAG Tool 工具边界，但可以返回 mock 结果。MVP 后将 mock 替换为 BM25 + 向量 + RRF 混合检索。

### 5.5 允许局部推倒重建

现有代码是第一轮原型，不是必须保留的架构约束。为了实现 AI 原生邮件工作台目标，可以重做前端结构、API 形状、Agent runtime 和部分数据模型。

应保留：

- 三服务形态。
- 邮件原文和用户邮箱条目分离的思想。
- 后端作为业务和权限中心。
- Agent 通过工具/API 访问业务数据。

允许重做：

- 当前单文件 `App.vue`。
- 旧式 mailbox CRUD API。
- 当前规则 Agent loop 的主结构。
- 静默 mock fallback。

## 6. 技术栈

### 6.1 前端

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 框架 | Vue 3 | AI 邮件工作台 |
| 构建 | Vite | 开发和构建 |
| 语言 | TypeScript | 约束 API 和组件数据 |
| UI | Element Plus | 表单、上传、抽屉、消息提示 |
| 状态 | Pinia | 用户、工作区、邮件列表、Agent 会话 |
| 路由 | Vue Router | 登录页和工作区路由 |
| HTTP | Axios | 调用后端 Workspace API |

前端重点不是堆页面，而是做出工作台体验：

- 左侧 Workspace Navigation。
- 中栏邮件列表和智能视图。
- 右侧 Mail Detail Drawer。
- Compose 面板、本地草稿自动保存、附件上传进度。
- 当前邮件 Agent 与全局 Agent 入口。

### 6.2 后端

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 语言 | Java 17 | 稳定运行环境 |
| 框架 | Spring Boot 3 | 主业务服务 |
| 安全 | Spring Security + JWT | 登录鉴权和接口保护 |
| ORM | MyBatis-Plus | 数据访问和分页 |
| 数据库 | MySQL 8，开发可用 H2 | 用户、邮件、分类、AI 任务和结果 |
| 构建 | Maven | 依赖管理和打包 |
| 可选 | Redis | 通知、缓存、限流等增强 |

后端模块建议：

- `auth`：注册、登录、鉴权。
- `user`：用户资料、设置、AI 写权限开关。
- `workspace`：工作台视图 API。
- `mail`：邮件原文、发送、详情。
- `mailbox`：用户邮箱条目、文件夹、软删除。
- `draft`：服务器草稿同步扩展点。
- `attachment`：Pending Attachment、正式附件、上传下载。
- `category`：用户类别和邮件分类结果。
- `analysis`：自动分析任务、状态、结果。
- `agent`：交互式 Agent 会话和消息。
- `internal`：Agent 使用的内部工具 API。
- `common`：响应、异常、安全上下文、工具类。

### 6.3 Agent 服务

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 语言 | Python 3.10+ | Agent 服务 |
| 框架 | FastAPI | 提供 Agent 和分析接口 |
| 模型定义 | Pydantic | 请求、响应、工具参数 |
| HTTP | httpx | 调用后端 Internal Tool API |
| Orchestration | LangChain，可用于后续 Tool Router/RAG | 工具路由和检索链路 |
| 主问答模型 | DeepSeek API | 当前邮件问答和后续全量问答 |
| Embedding | Ollama 本地小模型 | MVP 后的向量检索 |

MVP 中 LangChain 不应成为系统稳定性的单点。可以先保留轻量 tool router 接口和规则 fallback，逐步替换内部实现。

## 7. 推荐数据模型

应继续保留：

- `sys_user`
- `mail_message`
- `mail_recipient`
- `mailbox_item`
- `mail_attachment`
- `mail_ai_result`

建议新增或调整：

- `pending_attachment`：选择即上传、发送时绑定。
- `mail_category`：用户私有类别。
- `mail_category_assignment`：邮件与类别关系。
- `ai_analysis_task`：投递后自动分析任务。
- `ai_analysis_result`：摘要、分类、Junk、状态等结构化结果，可与现有 `mail_ai_result` 合并或演进。
- `agent_session`：交互式 Agent 会话。
- `agent_message`：Agent 对话消息和工具调用记录。
- `user_setting`：用户偏好、Agent 自动写权限、AI 开关。

草稿第一版以本地自动保存为主。若 pending attachment 需要服务端 compose session，可增加轻量 `compose_session`，但不要让未发送草稿混入正式 `mail_message`。

## 8. API 设计原则

面向前端主体验的 API 使用 Workspace View Model，不暴露数据库表拼装压力。

推荐 API 族：

- `GET /api/v1/workspace/views`
- `GET /api/v1/workspace/mail-items?view=inbox`
- `GET /api/v1/workspace/mail-items/{itemId}`
- `POST /api/v1/compose/attachments`
- `DELETE /api/v1/compose/attachments/{pendingAttachmentId}`
- `POST /api/v1/mails/send`
- `PATCH /api/v1/mailbox/items/{itemId}/read`
- `PATCH /api/v1/mailbox/items/{itemId}/category`
- `DELETE /api/v1/mailbox/items/{itemId}`
- `POST /api/v1/agent/sessions`
- `POST /api/v1/agent/sessions/{sessionId}/messages`

详细接口草案见 [docs/api/mvp-api.md](./docs/api/mvp-api.md)。

## 10. 最后三个待确认的大问题

以下问题不阻塞当前方案，但会影响下一轮实现优先级。

1. AI 自动分析是否必须在收件箱列表中实时刷新？
   推荐答案：MVP 使用手动刷新或轻量轮询；SSE/WebSocket 放到增强阶段。

2. 全局 Agent 在 MVP 中是否需要执行写操作？
   推荐答案：不需要。MVP 中全局 Agent 主要展示入口、对话框架和 mock RAG Tool；写操作优先在当前邮件 Agent 中做用户确认。

3. UI 是否优先适配桌面端？
   推荐答案：优先桌面端 1366px 以上工作台体验；移动端只保证可用，不作为 MVP 评分主战场。
