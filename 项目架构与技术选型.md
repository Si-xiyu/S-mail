# SmartMail 项目架构与技术选型方案

本文基于 `doc/要求/2026年实训要求.pdf` 的课程需求，以及 `doc/要求/prompt/gpt.md`、`doc/要求/prompt/实现方案.md` 中已有的 GPT 技术方向整理，确定 SmartMail 的具体实现方式、项目架构和技术选型。

## 1. 需求结论

课程 PDF 对邮件系统的核心要求可以归纳为：

- 支持邮箱注册、登录和用户身份校验。
- 客户端可以发送邮件到服务器，服务器保存邮件数据。
- 客户端可以接收或拉取服务器中的邮件。
- 支持收件箱、已发送、附件、已读/未读、删除、搜索、同步等基础邮件能力。
- 服务端数据库设计需要合理，建议按 E-R 关系拆成多个表。
- 邮件正文和附件必须分离存储：正文入库，附件存磁盘文件，数据库保存附件元信息。
- 可以采用 SMTP/POP3，也可以基于 HTTP、TCP、WebSocket 自定义上层协议。
- 智能系统需要以可插拔形式融合到基础邮件系统中，可实现垃圾邮件识别、优先级预测、预测性推送、安全检测、摘要生成等能力。
- Web 服务推荐前后端分离，前端可使用 Vue，后端框架不限。

结合实训周期、团队协作难度和已有技术方向，最终推荐做 Web 版智能邮件系统，不做 Qt App，不优先实现真实 SMTP/POP3 协议。

## 2. 最终实现方式

### 2.1 总体决策

SmartMail 采用：

```text
Vue 3 前端
    ↓ HTTP REST API
Spring Boot 3 主业务后端
    ├─ MySQL
    ├─ 本地附件存储
    ├─ Redis，可选
    └─ Internal Tool API
            ↓ HTTP
Python FastAPI AI Agent 服务，可选启动
    ├─ 规则引擎
    ├─ LLM SDK
    └─ Tool Calling Loop
```

核心实现策略：

- 邮件传输协议：首版采用 HTTP REST 自定义上层邮件协议。
- 邮件投递范围：假设用户都在同一个服务器下，不考虑跨邮件服务器传递。
- 邮件投递方式：发送邮件时后端写入邮件主表，并为发件人、收件人生成各自的邮箱条目。
- 邮件接收方式：客户端通过列表接口主动拉取；加分项可增加 SSE 或 WebSocket 新邮件通知。
- AI 能力：作为可插拔增强服务，不影响基础邮件收发。
- 附件存储：文件保存到服务器本地目录，数据库只保存路径、原始文件名、MIME 类型、大小和校验信息。

### 2.2 为什么不优先实现真实 SMTP/POP3

PDF 允许使用 HTTP 自定义上层协议。对于实训项目，HTTP 方案更适合首版落地：

- 前后端联调简单，接口可直接用 Axios 调用。
- 便于实现登录态、权限校验、分页、搜索、附件上传下载。
- 避免在短周期内处理 SMTP/POP3 的状态机、认证、MIME 兼容和网络调试成本。
- 仍然可以在文档中保留协议扩展设计，后续增加 SMTP/POP3 Adapter。

因此本项目的定位不是“完整邮件协议服务器”，而是“同服务器用户之间的 Web 邮件系统 + 智能插件能力”。

## 3. 技术选型

### 3.1 前端

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 框架 | Vue 3 | 构建单页应用 |
| 构建工具 | Vite | 快速开发和打包 |
| 语言 | TypeScript | 约束接口和组件数据结构 |
| UI 组件 | Element Plus | 快速实现表单、表格、弹窗、上传等组件 |
| 路由 | Vue Router | 页面路由 |
| 状态管理 | Pinia | 管理用户、邮箱、筛选、AI 面板状态 |
| HTTP | Axios | 调用后端 REST API |

前端页面重点：

- 登录页、注册页。
- 主邮箱布局。
- 收件箱、已发送、草稿箱、垃圾箱。
- 邮件列表与邮件详情。
- 写邮件弹窗或独立页面。
- 附件上传下载。
- 搜索与筛选。
- AI 摘要、回复草稿、优先级提示。

### 3.2 后端

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 语言 | Java 17 | Spring Boot 3 稳定运行环境 |
| 框架 | Spring Boot 3 | 主业务后端 |
| 安全 | Spring Security + JWT | 登录鉴权和接口保护 |
| ORM | MyBatis-Plus | CRUD、分页、条件查询 |
| 数据库 | MySQL 8 | 存储用户、邮件、附件元信息、AI 结果 |
| 构建 | Maven | 依赖管理和打包 |
| 缓存 | Redis，可选 | 验证码、未读计数、通知状态、限流 |
| 文档 | springdoc-openapi，可选 | 接口文档和联调 |

后端是系统核心，必须在 AI 服务关闭时仍然完整可用。

### 3.3 AI 服务

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 语言 | Python 3.10+ | AI 服务开发 |
| Web 框架 | FastAPI | 对外提供 AI 能力接口 |
| 数据模型 | Pydantic | 定义请求、响应和工具参数 |
| HTTP 客户端 | httpx | 调用 Spring Boot Internal API |
| LLM SDK | OpenAI SDK 或 DeepSeek SDK | 调用大模型 |
| 任务模式 | Tool Calling Loop | 让 AI 调用受控业务工具 |

不建议首版引入 LangChain、AutoGen、CrewAI 等重型 Agent 框架。原因是抽象层较多，调试成本高，且会削弱课程项目中“自己设计插件化智能系统”的展示价值。

### 3.4 可选能力

| 能力 | 首版建议 | 说明 |
| --- | --- | --- |
| Redis | 可选 | 没有 Redis 也应能运行 |
| WebSocket/SSE 通知 | 第二阶段 | 用于新邮件通知 |
| 邮件全文索引 | 第二阶段 | 首版 MySQL LIKE + 索引即可 |
| 向量检索 | 第三阶段 | 用于自然语言智能搜索 |
| SMTP/POP3 Adapter | 第三阶段 | 作为协议扩展，不放入首版主链路 |

## 6. 数据库设计

### 6.1 核心设计原则

邮件系统建议区分“邮件原文”和“用户邮箱条目”：

- `mail_message` 保存一封邮件的原始内容。
- `mailbox_item` 保存某个用户视角下这封邮件处于哪个文件夹、是否已读、是否星标、是否删除。

这样可以同时支持：

- 发件人看到已发送。
- 收件人看到收件箱。
- 不同用户对同一封邮件有不同已读状态。
- 不同用户独立删除、标星、加标签。

## 8. 邮件业务流程

### 8.1 注册登录

1. 用户提交邮箱、用户名、密码。
2. 后端校验邮箱唯一性。
3. 密码 BCrypt 哈希后入库。
4. 登录成功后返回 JWT。
5. 前端保存 Token，并在后续请求中携带 `Authorization: Bearer <token>`。

### 8.2 发送邮件

1. 用户填写收件人、抄送、主题、正文和附件。
2. 前端先上传附件，拿到附件 ID。
3. 前端调用发送接口。
4. 后端校验收件人是否是本系统用户。
5. 后端写入 `mail_message`、`mail_recipient`、`mail_attachment`。
6. 后端为发件人生成 `SENT` 邮箱条目。
7. 后端为每个收件人生成 `INBOX` 邮箱条目。
8. 如果开启 AI 分析，异步提交分析任务。

### 8.3 拉取邮件

1. 前端请求 `/api/v1/mailbox?folder=INBOX`。
2. 后端按 `mailbox_item.user_id` 和 `folder` 查询。
3. 返回分页邮件列表，包括主题、发件人、摘要、时间、已读状态、优先级、附件标识。
4. 用户打开详情时，后端返回正文、附件和 AI 结果。

### 8.4 删除邮件

首版采用软删除：

- 从普通文件夹删除时，将 `folder` 改为 `TRASH`。
- 从垃圾箱删除时，将 `deleted_flag` 改为 true。
- 不直接删除 `mail_message`，避免影响其他用户邮箱视图。

### 8.5 搜索邮件

首版搜索范围：

- 主题。
- 发件人邮箱。
- 正文关键词。
- 标签。
- 文件夹。
- 已读/未读。
- 时间范围。

优先使用 MySQL 组合条件和分页。若后续需要更好的体验，再加入全文索引或向量检索。

## 9. AI 插件化设计

### 9.1 插件边界

AI 不是主业务系统的硬依赖。Spring Boot 提供统一 AI 门面：

```text
AiService
├─ DisabledAiProvider      # 关闭 AI 时的空实现
├─ RuleBasedAiProvider     # 规则实现，例如关键词垃圾邮件识别
└─ RemoteAgentAiProvider   # 调用 Python Agent
```

配置示例：

```yaml
smartmail:
  ai:
    enabled: true
    provider: remote-agent
    agent-base-url: http://localhost:8000
    summary-enabled: true
    spam-enabled: true
    priority-enabled: true
    reply-draft-enabled: true
```

如果 `enabled=false` 或 Agent 不可用，基础邮件功能照常运行。

### 9.2 Agent Tool Calling Loop

Agent 内部流程：

```text
用户任务或后端任务
    ↓
构造 messages
    ↓
LLM 判断是否需要工具
    ↓
调用 Spring Boot Internal API
    ↓
将工具结果追加到 messages
    ↓
LLM 生成最终结果
    ↓
回写 AI 结果到 Spring Boot
```

### 9.3 首版 AI 功能

首版建议只做 3 个最有展示价值且容易验收的能力：

1. 邮件摘要：根据邮件正文生成 3 到 5 条要点。
2. 回复草稿：根据原邮件生成可编辑回复。
3. 优先级与标签建议：输出 `priority` 和标签候选。

垃圾邮件识别可以先做规则版：

- 标题或正文包含高风险词。
- 发件人异常。
- 链接数量过多。
- 附件类型可疑。

之后再切换到 LLM 或传统机器学习模型。

### 9.4 同步与异步

AI 任务按实时性拆分：

| 任务 | 执行方式 | 原因 |
| --- | --- | --- |
| 垃圾邮件识别 | 邮件到达后同步或准实时 | 影响收件箱展示 |
| 优先级预测 | 邮件到达后异步 | 可稍后更新 |
| 摘要生成 | 打开详情时懒加载或异步 | 避免拖慢收件 |
| 回复草稿 | 用户点击时实时生成 | 用户主动触发 |
| 自然语言搜索 | 用户输入时实时生成 | 交互型任务 |

1. 启动 MySQL。

3. 启动 Vue 前端。

### 11.3 部署方式

首版推荐单机部署或 Docker Compose：

- 前端构建为静态文件，可由 Nginx 托管。
- 后端打成 Jar 运行。
- Agent 单独运行。
- MySQL 单独容器或本机服务。
- 附件目录挂载到宿主机持久化路径。

不建议首版使用 Kubernetes。

## 15. 最终技术路线

最终推荐路线：

- 实现形态：Web 邮件系统。
- 通信协议：HTTP REST 自定义上层邮件协议。
- 前端：Vue 3 + Vite + TypeScript + Element Plus + Pinia + Axios。
- 后端：Spring Boot 3 + Java 17 + MyBatis-Plus + MySQL 8 + JWT。
- AI：Python FastAPI + Pydantic + httpx + LLM SDK + 轻量 Tool Calling Loop。
- 存储：邮件正文入 MySQL，附件落磁盘，数据库保存附件元信息。
- 架构：前后端分离，主业务后端独立运行，AI 服务插件化接入。
- 首版目标：先完成稳定的邮件收发闭环，再做 AI 展示能力。

这个方案符合课程 PDF 中对邮件系统、数据库设计、正文附件分离、智能插件和 Web 架构的要求，同时也兼顾了实训周期内的可实现性和答辩展示效果。
