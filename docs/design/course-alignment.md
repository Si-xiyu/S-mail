# SmartMail 课程要求对齐说明

本文对照 `doc/2026年实训要求.pdf`，说明 SmartMail 当前目标方案如何覆盖课程硬性要求、智能系统要求和加分项，并列出需要补强的实现点。

## 1. 总体结论

当前方案与课程要求没有根本冲突。课程明确允许“不基于现有 SMTP/POP3 协议，可以基于 HTTP 实现，在上层自定义消息封装协议”，并假设客户端在同一个服务器下，不考虑服务器间传递。因此 SmartMail 第一版采用同服务器站内邮箱通信和 HTTP Workspace API 是合规的。

需要优化的是表达和实现优先级：

- 搜索、同步/自动拉取、附件、已读未读、删除应写成 MVP 必交付，而不是可选。
- AI 需要明确是“可插拔智能系统”，并提供开关、规则 fallback、大模型 API 配置。
- 垃圾邮件、风险检测、优先级预测应作为自动分析管道的核心能力。
- 优先级排序最好引入用户行为信号，至少记录打开、回复、星标、删除、Junk 等事件。
- 新邮件到达通知前移为 MVP，MVP 可用轮询、未读数和站内提示，后续增强为 SSE/WebSocket。

## 2. 邮件系统功能要求

| 课程要求 | 当前方案覆盖 | 对齐判断 | 优化建议 |
| --- | --- | --- | --- |
| 邮箱注册，服务器保存注册信息用于登录验证 | `auth`、`sys_user`、JWT | 已覆盖 | 答辩演示注册、登录、当前用户信息 |
| 客户端发送邮件到服务器 | `POST /api/v1/mails/send` | 已覆盖 | 展示收件人、抄送、主题、正文、附件 |

| 客户端接收/拉取服务器邮件 | Workspace Mail List API | 基本覆盖 | 明确实现登录后自动拉取、手动刷新、轻量轮询 |

## 3. 协议与报文要求

课程给出三种协议路线：

- 标准实现：SMTP/POP3。
- 协议扩展：底层 SMTP/POP3，上层自定义。
- 自定义协议：基于 TCP、WebSocket、HTTP 上层实现。

SmartMail 选择第三种：基于 HTTP 的自定义上层邮件协议。

为避免答辩时被误认为“没有做协议”，文档和演示应明确说明：

- 我们实现的是 HTTP 自定义应用层协议。
- 通过 REST 方法表达报文动作：`POST` 发送、`GET` 拉取、`PATCH` 更新状态、`DELETE` 删除。
- 邮件报文字段覆盖课程示例：发送方、接收者、抄送、主题、正文、附件信息、时间戳。
- 附件不直接塞进正文，文件存磁盘，数据库保存 MIME、大小、路径、摘要等元数据。

建议在答辩中使用“自定义 HTTP 邮件协议”而不是“没有使用邮件协议”。

## 4. 智能系统要求

| 课程要求 | 当前方案覆盖 | 对齐判断 | 优化建议 |
| --- | --- | --- | --- |
| 智能系统以可插拔插件形式融合到基础邮件服务器 | AI Plugin + FastAPI Agent + Internal Tool API + 可关闭配置 | 已强化 | 关闭后回到 Basic Mail Mode；开启后进入 AI Mail Workspace Mode |
| 垃圾邮件识别插件 | Junk Mail 分类 | 已覆盖 | 作为自动分析高优先级任务 |
| 邮件优先级排序和预测性推送 | priority 字段、Important 视图 | 需要强化 | 引入 Behavior Signal，基于打开/回复/星标等生成优先级 |
| 基于用户历史行为预测重要性 | 当前方案未充分强调 | 有扣分风险 | 新增 `mail_event` 或行为记录，用规则版先实现 |
| 安全检测，如伪造发件人、恶意链接 | `riskHints` 有草案 | 需要强化 | MVP 做链接和发件人风险提示 |
| DLL 动态链接库方式提供 ML 插件 | 课程说“可以” | 不冲突 | 可作为扩展，不作为 Web 版 MVP 必需 |
| 大模型服务 SDK，用户自行填写 API 开启 | DeepSeek/Ollama 规划 | 需要强化 | Settings 中增加 AI 开关和 API Key 配置说明 |
| 垃圾邮件过滤实时处理，摘要异步运行 | 自动分析管道 | 需要细化 | Junk/风险/优先级近实时，摘要可低优先级异步 |

## 6. 必须补强的方案项

### 6.1 Mail Sync 进入 MVP

MVP 应明确实现：

- 登录成功后自动加载 Inbox。
- 切换 View/Folder/Category 时从服务端拉取。
- 提供刷新按钮。
- 使用轻量轮询更新未读数、新邮件和分析状态。

SSE/WebSocket 保留为增强项。

### 6.2 搜索进入 MVP

MVP 搜索至少覆盖：

- 主题。
- 发件人。
- 正文。
- 分类。

实现上可以先用数据库 `LIKE` 和普通索引，不要求全文索引或向量检索。

### 6.3 行为信号与优先级预测

为满足“基于用户历史行为预测重要性”，建议新增行为记录：

- `OPENED`
- `REPLIED`
- `STARRED`
- `DELETED`
- `MARKED_JUNK`

MVP 可以使用规则评分：

- 频繁打开/回复的发件人加权。
- 包含截止时间、老师、项目等关键词加权。
- 被用户多次标为 Junk 的发件人降权。

这能在答辩时说明已经具备“用户历史行为驱动”的基础。

### 6.4 安全风险提示

MVP 应在自动分析结果中加入：

- 链接检测：正文包含 URL 时提示用户确认来源。
- 发件人风险：显示名和邮箱域名不一致、异常域名等。
- 风险词提示：中奖、贷款、验证码索取、转账等。

这不等于病毒扫描，也不自动删除邮件。

### 6.5 AI 插件化和开关

应明确三层 fallback：

1. Basic Mail Mode：关闭 AI Plugin 后回到基础邮件系统。
2. 规则引擎：开启 AI Plugin 但无 API Key 时，仍可演示垃圾识别、优先级、风险提示。
3. 大模型 Provider：用户填写 DeepSeek API Key 后启用摘要、问答等增强能力。
4. RAG Tool：MVP mock，后续接 BM25 + 向量 + RRF。

AI 失败时基础邮件系统必须可用。

## 7. 需要更新的文档和实现

文档层面：

- 主入口文档将搜索、Mail Sync、行为信号、风险提示列入 MVP。
- API 文档补搜索参数、行为事件、设置项和分析结果字段。
- 架构文档补自动分析任务优先级：Junk/风险/优先级近实时，摘要异步。

实现层面：

- 前端增加 Mail Sync 策略：登录加载、刷新、轮询。
- 后端增加搜索 API 或扩展 Workspace API 的 `keyword`。
- 后端增加行为事件记录。
- 自动分析结果增加 `riskHints`、`priorityScore`、`category`、`junk`。
- Settings 增加 AI 开关和 API Key 配置入口。

## 8. 答辩表述建议

推荐答辩时这样表述协议选择：

> 课程允许基于 HTTP 自定义上层协议实现邮件系统。我们的系统没有做跨服务器 SMTP/POP3，而是在同一服务器内实现完整邮件业务闭环：客户端通过 HTTP 协议向服务器发送、拉取和同步邮件，服务器负责用户认证、邮件持久化、附件隔离存储和智能分析。

推荐这样表述智能系统：

> 智能系统以可插拔 Agent 服务接入主业务后端。基础邮件系统不依赖 AI；AI 可关闭、可降级。垃圾邮件、风险检测和优先级预测在邮件到达后近实时处理；摘要和问答使用异步或用户触发方式，避免拖慢主链路。
