# SmartMail 项目实现方案

> 注意：本文是早期实现方案记录，保留用于追溯。当前目标方案以根目录 [项目架构与技术选型.md](../项目架构与技术选型.md)、[SmartMail项目全貌.md](../SmartMail项目全貌.md) 和 [CONTEXT.md](../CONTEXT.md) 为准。

本文基于 `doc/要求/2026年实训要求.pdf` 的需求约束，以及 `doc/要求/prompt/gpt.md` 中已经整理出的技术方向，进一步落到可直接实施的项目架构、模块划分和技术选型。

## 1. 项目定位

SmartMail 的目标不是做一个“纯邮件客户端”，而是做一个具备 AI 能力的现代化邮件系统。系统需要覆盖完整业务闭环，包括注册登录、邮件收发、附件管理、搜索、已读未读、标签/优先级、以及基于 Agent 的智能分析能力。

项目重点有三点：

1. 业务功能完整，能够演示“收、发、查、管、智能分析”的闭环。
2. AI 模块与主业务系统解耦，AI 失效时邮件系统仍能正常使用。
3. 采用工程化架构，便于后续扩展、答辩展示和功能迭代。

## 2. 需求拆解

### 2.1 核心业务功能

根据需求文档，系统至少要具备以下能力：

- 用户注册、登录、退出、Token 鉴权。
- 邮件发送、接收、草稿、已发送、收件箱管理。
- 邮件阅读状态管理，包括已读、未读、删除、回收站或软删除。
- 邮件搜索，支持按主题、发件人、正文关键词检索。
- 附件上传、下载、预览或基本信息展示。
- 标签、优先级、星标等基础整理能力。

### 2.2 AI 增强功能

AI 功能建议围绕邮件工作流来设计，而不是独立做一个聊天机器人。可落地的能力包括：

- 邮件摘要。
- 垃圾邮件识别。
- 邮件优先级分析。
- 自动标签分类。
- 回复草稿生成。
- 自然语言搜索邮件。

### 2.3 设计约束

需求中有几个关键约束需要遵守：

- AI 模块要低耦合，系统关闭 AI 后仍可正常使用。
- Python Agent 不直接访问数据库，所有业务动作必须通过 Spring Boot 的内部 API 完成。
- 邮件正文存数据库，附件存文件系统，数据库只保留附件元信息。
- 技术路线以 Web 端为主，更符合当前给出的 Vue + Spring Boot + Python 方向。

## 3. 最终实现方式

我建议采用“前后端分离 + 主业务后端 + 独立 AI Agent 服务”的三层实现方式：

1. 前端负责邮件产品交互，使用 Vue 3 构建类似 Gmail / Notion Mail 的体验。
2. Spring Boot 负责全部业务主链路，包括用户、邮件、附件、标签、搜索、内部 API。
3. Python FastAPI 负责 AI 能力，通过 Tool Calling 方式调用 Spring Boot 内部接口，再把 AI 结果回写到业务系统。

这个方案的优点是边界清楚：

- 业务系统独立可运行。
- AI 能力可开关。
- 后续可以替换 LLM、规则引擎或向量检索实现，不影响主系统。

## 4. 系统架构

### 4.1 逻辑架构

```text
Vue 3 前端
    ↓ REST API
Spring Boot 3 主业务后端
    ├─ MySQL
    ├─ 本地附件存储
    ├─ Redis（可选）
    └─ Internal API
            ↓ HTTP
Python FastAPI Agent 服务
    ├─ Tool Calling Loop
    ├─ 规则引擎
    ├─ LLM SDK
    └─ 向量检索（可选）
```

### 4.2 服务职责边界

#### 前端

- 登录注册页。
- 左侧邮箱列表、分类、标签视图。
- 收件箱、已发送、草稿、垃圾箱等页面。
- 邮件详情页。
- 写信页。
- AI 助手面板。
- 搜索框和过滤器。

#### Spring Boot 主业务后端

- 用户认证与授权。
- 邮件收发和状态流转。
- 附件上传与下载。
- 搜索与索引。
- 标签、优先级、星标、已读未读。
- AI 结果存储。
- 对外 REST API 和对内 Internal API。

#### Python AI Agent

- 读取用户输入或任务事件。
- 调用 LLM 判断是否需要工具。
- 通过 Tool Calling 调用内部 API。
- 生成摘要、草稿、标签建议、优先级建议。
- 将 AI 结果提交回主业务后端。

## 5. 技术选型

### 5.1 前端技术栈

推荐采用：

- Vue 3
- Vite
- TypeScript
- Element Plus
- Pinia
- Vue Router
- Axios

选择理由：

- Vue 3 上手快，适合团队协作和中型项目落地。
- Element Plus 适合快速搭建邮箱类管理界面。
- TypeScript 可以降低组件和接口联调错误。
- Pinia 适合管理邮箱状态、筛选状态、会话状态。

### 5.2 后端技术栈

推荐采用：

- Java 17 或 21
- Spring Boot 3
- MyBatis-Plus
- MySQL 8
- Spring Security + JWT
- Maven
- Redis（可选）

选择理由：

- Spring Boot 3 适合构建稳定的业务后端。
- MyBatis-Plus 更适合 CRUD 密集型邮件业务。
- MySQL 足够支撑实训项目的数据规模。
- JWT 便于前后端分离鉴权。
- Redis 可用于验证码、缓存、未读计数和简单限流，但不作为核心依赖，避免复杂度过高。

### 5.3 AI 服务技术栈

推荐采用：

- Python 3.10+
- FastAPI
- Pydantic
- httpx
- OpenAI SDK 或 DeepSeek SDK

选择理由：

- FastAPI 适合快速实现轻量 AI 服务。
- Pydantic 便于定义 Tool 输入输出结构。
- httpx 适合调用内部 API 和第三方 LLM API。
- 轻量 Tool Calling Loop 比重型 Agent 框架更可控，也更符合需求里“业务解耦”的要求。

## 6. 后端项目架构建议

### 6.1 模块划分

Spring Boot 项目建议按以下模块拆分：

- `auth`：登录、注册、JWT、权限控制。
- `user`：用户信息、邮箱账户设置、个人配置。
- `mail`：邮件收发、列表、详情、状态变更。
- `draft`：草稿保存和恢复。
- `attachment`：上传、下载、文件元数据管理。
- `label`：标签管理、自动/手动标注。
- `search`：关键词检索、组合查询。
- `ai`：AI 结果存储、内部任务状态、开关控制。
- `common`：统一返回、异常处理、工具类。

### 6.2 推荐分层

建议采用标准三层：

- Controller：接收 HTTP 请求，做参数校验和接口编排。
- Service：处理业务规则和事务。
- Repository/Mapper：负责数据持久化。

对于邮件系统这种以业务流程为中心的项目，这种分层比过度 DDD 更容易控制复杂度，也更适合实训周期。

### 6.3 内部 API 设计原则

Python Agent 不能直连数据库，因此 Spring Boot 需要提供专用内部接口，例如：

- `search_mail(keyword)`
- `get_mail(mail_id)`
- `apply_label(mail_id, label)`
- `draft_reply(mail_id)`
- `mark_as_read(mail_id)`

这些接口建议放在单独的 Internal Controller 中，并通过以下方式保护：

- 内网访问或网关隔离。
- 简单签名或服务间 Token。
- 仅对 AI 服务开放必要能力。

## 7. 数据设计建议

### 7.1 核心表

建议至少包括以下实体：

- `user`：用户信息、账号状态。
- `mail`：邮件主表，保存主题、正文、发件人、收件人、时间、状态。
- `mail_recipient`：收件人关联表，支持一封邮件多个收件人。
- `mail_attachment`：附件元数据表。
- `mail_label`：邮件标签表或标签定义表。
- `mail_label_rel`：邮件与标签关系表。
- `mail_read_status`：已读未读状态。
- `mail_ai_result`：AI 摘要、分类、优先级、草稿等结果。

### 7.2 存储策略

- 邮件正文直接存 MySQL。
- 附件文件存本地磁盘或独立文件目录。
- 数据库只保存附件路径、文件名、MIME 类型、大小等元数据。

这种方式实现简单，便于演示，也符合需求中“数据库只存元信息”的方向。

### 7.3 搜索策略

实训项目建议优先使用 MySQL 模糊搜索和组合索引，必要时再加全文索引。

如果需要展示更强的 AI 搜索能力，可以在后续加入：

- 邮件内容向量化。
- 向量检索服务。
- 结合 LLM 做自然语言查询解析。

但在首版实现里，不建议一开始就引入过重的搜索基础设施。

## 8. AI Agent 实现方式

### 8.1 推荐实现模式

采用轻量 Tool Calling Loop：

```text
messages -> LLM -> tool_calls? -> execute_tool -> append result -> LLM -> final_answer
```

### 8.2 为什么不用重型框架

不建议直接使用 LangChain、AutoGen、CrewAI 之类框架，原因是：

- 对实训项目来说，抽象层太多，调试成本高。
- Tool 调用和状态流转反而会变复杂。
- 需求更强调“能落地、可控、易展示”，而不是搭一个复杂 Agent 平台。

### 8.3 AI 能力落点

AI 建议绑定到具体邮件动作上：

- 进入邮件详情页时自动生成摘要。
- 新邮件进入时做垃圾邮件或优先级提示。
- 用户点击“生成回复草稿”时调用 Agent。
- 搜索框支持自然语言输入，Agent 转换成检索条件。

### 8.4 AI 开关设计

AI 模块必须支持关闭，建议通过配置控制：

```yaml
ai:
  enable_summary: true
  enable_spam_filter: true
  enable_priority: false
```

如果 AI 服务不可用，前端展示普通邮件能力，不影响收发、搜索和管理功能。

## 9. 业务流程建议

### 9.1 发送邮件流程

1. 用户在前端编写邮件并提交。
2. 前端调用 Spring Boot 发送接口。
3. 后端保存邮件、附件元数据和状态。
4. 如需 AI 分析，则异步提交给 Python Agent。
5. Agent 返回标签建议、摘要或优先级建议。
6. 后端持久化 AI 结果，前端展示结果。

### 9.2 阅读邮件流程

1. 前端打开邮件详情。
2. 后端返回邮件正文、附件信息和 AI 结果。
3. 后端同步标记已读。
4. 可选触发摘要生成或阅读行为统计。

### 9.3 智能搜索流程

1. 用户输入自然语言，例如“找出上周客户发来的催办邮件”。
2. Python Agent 解析意图。
3. Agent 调用 Spring Boot 内部搜索接口。
4. 后端返回候选邮件列表。
5. Agent 可进一步总结结果并返回给前端。

## 10. 前端页面架构建议

### 10.1 页面结构

- 登录/注册页。
- 主布局页。
- 邮箱分类页。
- 邮件列表页。
- 邮件详情页。
- 写信页。
- AI 助手侧边栏。
- 用户设置页。

### 10.2 组件建议

- 邮件列表卡片。
- 邮件阅读面板。
- 标签选择器。
- 附件上传器。
- AI 摘要卡片。
- 回复草稿编辑器。

### 10.3 UI 风格

整体风格参考 Gmail、Outlook、Notion Mail，但建议偏向“简洁、信息密度高、可快速处理邮件”的工作台风格，而不是过度消费级设计。

## 11. 部署建议

### 11.1 本地开发结构

建议本地开发时拆成三个可独立启动的服务：

- `frontend`
- `backend`
- `agent`

### 11.2 推荐端口

- 前端：`5173`
- 后端：`8080`
- AI 服务：`8000`

### 11.3 部署方式

实训阶段建议优先单机部署或 Docker Compose 部署，避免一开始上 Kubernetes。

原因很简单：

- 交付周期短。
- 组件少。
- 更容易演示和调试。

## 12. 推荐的第一版落地范围

为了保证可以按期完成，第一版建议优先实现以下内容：

1. 用户注册登录与 JWT 鉴权。
2. 邮件收件箱、已发送、邮件详情、写信、附件上传下载。
3. 邮件搜索、已读未读、标签、优先级基础能力。
4. Python Agent 的摘要和草稿生成功能。
5. AI 开关和内部 API 访问控制。

第二阶段再补充：

- 更完善的自然语言搜索。
- 垃圾邮件识别。
- 自动标签分类。
- 统计分析面板。

## 13. 最终结论

综合需求文档和现有技术方向，最合适的实现方案是：

- 前端：Vue 3 + Vite + TypeScript + Element Plus。
- 主业务后端：Spring Boot 3 + MyBatis-Plus + MySQL + JWT。
- AI 服务：Python FastAPI + 轻量 Tool Calling Loop + LLM SDK。
- 架构：前后端分离 + AI 微服务解耦 + 内部 API 驱动。
- 数据策略：正文入库，附件落盘，AI 结果单独存储。

这个方案兼顾了实训落地性、工程化结构和 AI 展示效果，适合作为 SmartMail 的主实现路线。
