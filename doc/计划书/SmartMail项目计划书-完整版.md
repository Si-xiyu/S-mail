# 1. 项目背景、范围及目标

## 1.1 项目背景

SmartMail 智能邮件系统是依据 2026 年软件开发综合实训“邮件系统”题目要求规划和开发的 Web 应用项目。课程要求系统至少具备邮箱注册、用户登录、发送邮件到服务器、从服务器接收或拉取邮件、服务器持久化邮件数据等基础能力，同时要求邮件正文和附件分离存储，并鼓励在基础邮件服务器中以可插拔方式融合智能系统能力。

传统邮件系统通常依赖 SMTP、POP3 或 IMAP 协议，但实训要求允许基于 HTTP、WebSocket 或 TCP 自定义上层协议实现邮件收发。结合团队开发周期、成员技术栈、前后端协同成本和答辩展示效果，本项目采用 Web 版前后端分离架构，以 HTTP REST API 作为首版通信协议，在同一服务器用户之间实现邮件收发闭环，并通过独立 Python Agent 服务提供邮件摘要、回复草稿、优先级分析和标签建议等智能能力。

截至本计划书编制时，项目仓库已完成第一轮最小 MVP：前端、后端、Agent 三部分均已搭建 mock 骨架；后端已建立 Spring Boot、数据库 schema、基础实体、Mapper、认证、邮件收发、邮箱列表、邮件详情、AI 门面接口和 Internal Tool API；Agent 已完成 FastAPI 服务、任务接口、规则版 Agent Loop、后端工具调用和 mock fallback；前端已完成类 Notion Mail 的工作台骨架，可调用后端 API，也可在后端不可用时使用 mock 数据。

## 1.2 项目意义

本项目不是单纯实现一个页面演示型邮箱，而是围绕“可运行的业务系统 + 可插拔智能能力 + 可协作工程结构”三个目标展开。项目的实践意义主要体现在：

1. 完整覆盖全栈开发流程：前端 Vue、后端 Spring Boot、数据库 MySQL、Python Agent 和接口联调均纳入统一工程。
2. 训练系统架构设计能力：通过前后端分离、业务后端和 Agent 解耦、Internal API、安全边界和数据分层设计，形成可解释、可扩展的系统结构。
3. 满足课程对数据库设计的要求：邮件主数据、收件人关系、用户邮箱视图、AI 结果等拆分为多张表，避免把复杂业务压缩到单表中。
4. 支持智能系统加分项：Agent 以服务化、可开关、可降级方式接入，不影响基础邮件系统可用性。
5. 便于团队并行开发：前端、后端、Agent 和文档分别有清晰目录和职责边界，适合 6 人小组分模块推进。

## 1.3 项目范围

### 1.3.1 首版范围

首版 SmartMail 的范围是“同一服务器用户之间的 Web 邮件系统 + 基础智能邮件处理能力”。系统假设所有用户均注册在同一服务器中，不处理跨邮件服务器转发，不接入真实外部邮箱，也不优先实现完整 SMTP、POP3、IMAP 协议栈。

首版包含以下范围：

- 用户注册、登录、退出和 Token 鉴权。
- 发送邮件、收取邮件、拉取邮箱列表、查看邮件详情。
- 收件箱、已发送、垃圾箱等基础邮箱视图。
- 已读/未读、星标、删除到垃圾箱、软删除等状态管理。
- 邮件正文入库，附件文件落盘，附件元信息入库。
- AI 邮件摘要、回复草稿、优先级分析、标签建议和垃圾邮件风险提示。
- 后端 AI 门面接口和 Agent Internal Tool API。
- 前端类 Notion Mail 的工作台式交互界面。
- 项目计划、接口、设计、测试和报告文档。

### 1.3.2 暂不纳入首版范围

为保证实训周期内交付稳定，本项目首版暂不承诺以下内容：

- 不实现跨邮件服务器通信。
- 不直接对接真实 SMTP/POP3/IMAP 服务器。
- 不实现企业级全文搜索集群或向量数据库。
- 不实现复杂多租户、组织通讯录和管理员后台。
- 不把 AI 服务作为基础邮件功能的强依赖。
- 不在首版引入 Kubernetes 等重型部署方式。

上述能力可作为第二阶段扩展或答辩中的演进设计说明。

## 1.4 系统总体架构

系统采用三服务架构：

| 层级 | 技术 | 职责 |
| --- | --- | --- |
| 前端 | Vue 3、Vite、TypeScript、Element Plus、Pinia、Axios | 邮件工作台、登录注册、邮箱列表、详情阅读、写信、AI 面板 |
| 主业务后端 | Spring Boot 3、Java 17、MyBatis-Plus、MySQL 8、Token 鉴权 | 用户、邮件、邮箱状态、附件、搜索、标签、AI 门面、Internal Tool API |
| AI Agent | Python、FastAPI、Pydantic、httpx、规则引擎、LLM SDK 预留 | 邮件摘要、回复草稿、优先级分析、标签建议、工具调用和 AI 结果写回 |

逻辑链路如下：

前端 Vue 工作台调用 Spring Boot REST API；Spring Boot 负责业务规则和数据库持久化；当用户触发 AI 任务时，后端调用 Python Agent；Agent 通过 Internal Tool API 获取邮件上下文、写回 AI 结果、更新邮件优先级。Agent 不直连数据库，前端也不直接调用 Agent。

## 1.5 系统基本功能

### 1.5.1 基础邮件功能

| 功能模块 | 功能说明 |
| --- | --- |
| 用户认证 | 邮箱注册、登录、Token 签发、当前用户查询 |
| 邮件发送 | 发件人填写收件人、抄送、主题、正文和附件，后端保存邮件并生成邮箱条目 |
| 邮件接收 | 收件人通过收件箱接口拉取邮件，后续可扩展 SSE 或 WebSocket 新邮件通知 |
| 邮箱管理 | 收件箱、已发送、垃圾箱、已读/未读、星标、删除 |
| 附件管理 | 附件上传、下载、磁盘隔离存储、数据库保存元信息 |
| 搜索与筛选 | 按主题、正文、发件人、文件夹、已读状态和时间范围查询 |
| 标签管理 | 用户自定义标签、AI 标签建议、邮件标签关系 |

### 1.5.2 智能邮件功能

| 智能能力 | 实现方式 | 价值 |
| --- | --- | --- |
| 邮件摘要 | Agent 获取邮件上下文后生成结构化摘要和行动项 | 降低阅读长邮件成本 |
| 回复草稿 | 根据邮件主题、发件人和正文生成可编辑回复 | 提高邮件处理效率 |
| 优先级分析 | 基于关键词、任务紧急性、风险词等规则判断优先级 | 帮助用户优先处理重要邮件 |
| 标签建议 | 识别课程、会议、附件、待处理等标签 | 改善邮件归类和检索体验 |
| 垃圾邮件识别 | 首版规则识别，后续可接入 LLM 或机器学习模型 | 增强邮件安全性 |

## 1.6 要解决的核心问题

| 问题 | 设计方案 | 预期效果 |
| --- | --- | --- |
| 邮件原文和用户状态混杂 | `mail_message` 保存邮件主数据，`mailbox_item` 保存用户视图状态 | 支持不同用户独立已读、删除、星标和优先级 |
| 附件不适合直接入库 | 文件存磁盘，数据库保存路径、文件名、MIME、大小和校验信息 | 降低数据库压力，符合课程要求 |
| AI 服务不稳定可能影响主链路 | 后端提供 AI 门面和降级返回，Agent 可选启动 | AI 失效时基础邮件功能仍可用 |
| 前后端联调周期短 | 先定义 REST API 和 mock 数据，再逐步切换真实接口 | 降低协作阻塞 |
| Agent 可能越权访问数据 | Agent 不直连数据库，只通过 Internal API 和 Internal Token 操作 | 控制智能系统权限边界 |
| SMTP/POP3 状态机复杂 | 首版采用 HTTP REST 自定义协议 | 保证实训周期内可落地、可演示 |

# 2. 项目可行性分析

## 2.1 经济可行性

SmartMail 采用开源技术栈构建，核心开发成本主要来自团队人力和时间投入。Vue、Spring Boot、MyBatis-Plus、MySQL、FastAPI、Pydantic、httpx 等均为开源技术，无需商业授权。AI 能力首版可采用规则引擎实现，后续如接入 DeepSeek、OpenAI 等大模型服务，也可通过开关控制调用频率，避免 API 成本失控。

| 成本项 | 估算 | 说明 |
| --- | --- | --- |
| 开发工具 | 0 元 | IDE、Git、Maven、Node、Python 均可免费使用 |
| 数据库 | 0 元 | MySQL 开源免费，本地开发可使用 H2 |
| 服务器 | 0-100 元 | 实训演示可使用本机、学校服务器或轻量云主机 |
| AI API | 0-100 元 | 首版规则引擎可零成本；大模型调用可按需少量开启 |
| 文档工具 | 0 元 | Markdown、Word、Draw.io 或类似工具 |

结论：项目经济成本低，投入主要为团队开发时间，经济可行性强。

## 2.2 技术可行性

本项目采用成熟、低风险、适合教学交付的技术栈。

| 技术 | 用途 | 成熟度 | 评估 |
| --- | --- | --- | --- |
| Vue 3 + Vite | 前端单页应用 | 高 | 构建快，生态成熟，适合邮箱工作台界面 |
| Element Plus | UI 组件 | 高 | 表单、表格、弹窗、上传组件齐全 |
| Spring Boot 3 | 主业务后端 | 高 | 企业级成熟框架，适合 REST API 和数据库业务 |
| MyBatis-Plus | ORM/Mapper | 高 | 适合 CRUD 密集型业务和分页查询 |
| MySQL 8 | 关系数据库 | 高 | 满足用户、邮件、附件、AI 结果等结构化存储 |
| FastAPI | Agent 服务 | 高 | Python Web 服务开发效率高，类型提示清晰 |
| Pydantic | Agent 数据模型 | 高 | 适合定义工具输入输出和 API schema |
| httpx | Agent 工具调用 | 高 | 适合调用 Spring Boot Internal API |

当前仓库已经验证了第一轮 MVP 的技术路径：后端 `mvn test` 通过，Agent `python -m compileall app` 通过，前端 `npm run build` 通过，说明三服务工程骨架具备继续扩展的基础。

## 2.3 运行可行性

系统采用单机多服务方式即可运行，推荐端口如下：

| 服务 | 端口 | 运行方式 |
| --- | --- | --- |
| 前端 Vite | 5173 | `npm run dev` |
| Spring Boot 后端 | 8080 | `mvn spring-boot:run` |
| Python Agent | 8000 | `uvicorn app.main:app` |
| MySQL | 3306 | 本机或容器 |

开发期可使用 H2 数据库启动后端，降低配置门槛；正式演示时切换 MySQL，体现数据库持久化能力。Agent 支持 mock fallback，即后端或 Internal API 暂不可用时仍可用 mock 邮件完成演示，避免因单点故障影响整体进度。

## 2.4 团队可行性

项目按 6 人小组设计，任务边界清晰：

| 角色 | 人数 | 主要职责 |
| --- | --- | --- |
| 组长/架构师 | 1 | 需求拆解、架构设计、接口协调、集成、答辩 |
| 后端开发 | 2 | Spring Boot、数据库、邮件收发、附件、搜索、标签 |
| 前端开发 | 2 | Vue 工作台、列表、详情、写信、AI 面板 |
| AI/测试 | 1 | Agent、智能功能、测试文档、演示数据 |

该分工可以并行推进。后端先给出 API 契约，前端用 mock 数据开发；Agent 通过 Internal API 契约独立开发；组长负责分支合并和集成测试。

## 2.5 风险可行性分析

| 风险 | 概率 | 影响 | 应对措施 |
| --- | --- | --- | --- |
| 前后端接口不一致 | 中 | 高 | 先维护 `docs/api`，接口变更必须同步文档 |
| 数据库表设计返工 | 中 | 高 | 保持邮件原文和用户视图分离，提前评审 ER 图 |
| Agent 接入阻塞 | 中 | 中 | Agent 使用 mock fallback，后端提供稳定 Internal API |
| LLM 服务不稳定 | 中 | 中 | 首版规则引擎可独立运行，大模型作为增强项 |
| 附件安全问题 | 中 | 中 | 使用 UUID 路径、限制文件类型、数据库只保存受控相对路径 |
| 时间不足 | 中 | 高 | 严格按 MVP 优先级交付，先主流程后加分项 |
| 集成测试延期 | 中 | 高 | 每周至少一次合并测试，提前准备演示账号和数据 |

综合结论：项目经济、技术、运行和团队层面均具备可行性；主要风险集中在接口协同、数据库设计和集成节奏上，均有明确应对方案。

# 3. 项目实施计划

## 3.1 开发策略

项目采用“先闭环、后增强”的实施策略：

1. 第一优先级是基础邮件系统闭环，包括用户、邮件、邮箱条目和数据库持久化。
2. 第二优先级是前端可交互工作台，使系统可以演示真实用户流程。
3. 第三优先级是 Agent 和 AI 能力，先用规则版实现稳定演示，再接入 LLM。
4. 第四优先级是附件、搜索、标签、通知、自然语言搜索等增强项。

分支策略采用模块分支：

| 分支 | 用途 |
| --- | --- |
| `main` | 主分支，保持可演示状态 |
| `backend/dev` | 后端开发与数据库流转 |
| `frontend/dev` | 前端页面与交互 |
| `agent/dev` | Agent、规则引擎、工具调用 |
| `docs/dev` | 文档、计划书、接口说明 |

## 3.2 阶段划分

项目计划周期为 5 周，保留第 6 周作为答辩缓冲和问题修复期。

| 阶段 | 周期 | 重点任务 | 主要交付物 |
| --- | --- | --- | --- |

| 阶段二：后端基础闭环 | 第 2 周 | 用户认证、邮件发送、收件箱、已发送、详情、数据库持久化 | Auth/Mail/Mailbox 后端接口、schema.sql |
| 阶段三：前端工作台与基础联调 | 第 3 周 | 登录注册、邮箱列表、邮件详情、写信、状态操作 | Vue 工作台、Axios API、mock fallback |

## 3.5 当前基线与后续推进

当前仓库已完成第一轮 MVP，已经具备以下基线：

- 后端：Spring Boot Maven 工程、schema、认证、邮件、邮箱、AI 门面、Internal Tool API。
- Agent：FastAPI、`/api/v1/agent/tasks`、规则版 Agent Loop、`get_mail`、`save_ai_result`、`set_priority` 工具调用。

后续推进应避免重新推翻架构，而是在当前边界内细化：

1. 后端补齐附件上传下载、搜索和标签 CRUD。
2. Agent 增加 LLM Provider，但保留规则引擎 fallback。
3. 前端将 `App.vue` 拆分为布局、列表、阅读器、写信、AI 面板组件。
4. 补充集成测试脚本和演示数据。
5. 把文档从“计划和接口”扩展为“设计、实现、测试、部署、使用说明”。

## 4.6 计划结论

SmartMail 项目采用 Web 前后端分离、Spring Boot 主业务后端和 Python Agent 智能服务的架构，符合课程对邮件系统、数据库设计、正文附件分离和智能插件的要求。当前仓库已完成第一轮 MVP，为后续功能细化提供了稳定基线。后续工作应围绕附件、搜索、标签、LLM 接入、测试和答辩材料展开，确保先交付稳定主流程，再展示智能增强能力。

从项目架构角度看，本计划的关键不是追求协议或 AI 框架复杂度，而是在实训周期内构建一个边界清晰、能运行、能演示、能解释的系统。该路线技术风险可控，团队分工明确，成果可验证，适合作为 SmartMail 项目的正式实施计划。
